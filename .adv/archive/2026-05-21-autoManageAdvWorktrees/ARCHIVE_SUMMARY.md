## Wisdom Accumulated

- **[gotcha]** Temporal worker non-determinism hits the `adv_run_test` signal path even when the change workflow itself is healthy. When the worker has stale `WorkflowTaskFailedCauseNonDeterministicError` on a different workflow's history, it can poison the `taskRunSummary` / activity-machine for unrelated change workflows too. Workaround that worked: terminate the change workflow via `temporal workflow terminate`, then reset to event ID 3 (`temporal workflow reset --event-id 3`) so it replays from a fresh state through whatever signals get re-applied. Going forward, prefer running tests via `pnpm vitest run <file>` in bash for the duration of the affected session and capture evidence in task-completion notes manually instead of `adv_run_test`. The classifier in `plugin/src/temporal/retry-wrapper.ts classifyTemporalError` does NOT match terminated-workflow errors as "fallback" — only TMPRL1100 / Nondeterminism / not_found patterns trigger re-seed. A future improvement would be to expand that classifier to include "terminated" / "workflow execution terminated".
- **[pattern]** When adding a new field to ChangeSchema that must be stamped at change creation, three surfaces need synchronized updates: (1) `ensureChangeWorkflowStarted` seedState so the workflow starts with the value, (2) `changeWithOwner` projection passed to `legacy.changes.save` so the disk snapshot has it on first save, (3) `updateOverlay` patch so Memo cache summary reads see it without a workflow round-trip. The same pattern applies to closure-time updates (look at `archived` status flow). For LAZY migration of legacy values, fire from `getTemporalChange` AFTER both the happy-path query AND the post-reseed query, checking both state.field AND the recently-fetched disk projection — never from the bare `loadChange` because that's pure-disk and lacks Temporal client access.
- **[convention]** For void-async fire-and-forget patterns in tests, `setImmediate × 50 + setTimeout 50ms` is the reliable settle window when the IIFE awaits `getGuardedChangeHandle` (which does a legacy disk read) before signaling. A single `setImmediate` is not enough because of the multi-hop `await input.legacy.changes.get(changeId)` → `getChangeHandle` → `handle.signal` chain.
