## Wisdom Accumulated

- **[pattern]** For public JSON schema artifacts, keep a curated registry of exported Zod schemas and generate deterministic files by sorting JSON object keys after `z.toJSONSchema()`. This avoids filesystem scraping, keeps Zod authoritative, and lets `schemas:check` compare committed artifacts byte-for-byte in CI.
- **[pattern]** When retiring published schema URLs, update both runtime scaffolding constants and tracked `.adv/specs/*/spec.json` refs, then add an asset test that scans stored artifacts plus scaffold defaults. Keep test-file mentions of the retired host allowed only as the sentinel string under test.
- **[pattern]** For tool preflight drift guards, export read-only policy metadata from the pure preflight module and test it two ways: an explicit audited-field matrix for required policy behavior, plus a registry-backed dead-entry check against `createToolMap()` args. This catches both missing audit policies and stale renamed fields without heuristic ownership.
- **[gotcha]** `pnpm test -- <file>` can still route through broader Vitest discovery in this repo and surface unrelated integration timeouts. For focused single-file evidence after such a mismatch, rerun with `pnpm exec vitest run <file>` and leave full-suite coverage to the final verification task.
- **[success]** Sub-agent report contract surfaces already have layered tests: schema strictness in `types/subagent-reports.test.ts`, ingest/persistence rejection in `tools/subagent-report.test.ts`, and packet/prompt/spec anchors in agent/spec asset tests. For preservation tasks, focused verification can be sufficient when no touched report contract code changes are needed.
- **[pattern]** When schema-generation behavior changes, update both agent-facing context (`project.md`) and developer quick-reference (`AGENTS.md`) plus asset tests that reject stale guidance. Also scan ignore/linguist comments (`.lgrepignore`, `.gitattributes`) because they often preserve old "stub/generated" assumptions outside docs.
- **[success]** For final contract hardening verification, one targeted suite can cover acceptance-specific surfaces before full gates: schema registry, schema URL guards, preflight policy, registry blocking, strict sub-agent reports, agent/spec asset anchors, and repo instruction drift. Follow with `pnpm run check`, `bin/oc-test full`, and `pnpm run build` for repo-level confidence.
