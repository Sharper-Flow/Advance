Architecture & State Management

Feature Goost ADV Assessment
State Storage In-memory plugin state + contract text parsing from AI responses. No persistent storage. JSON files as source of truth + SQLite with FTS5 for fast search. Full persistence across sessions. ADV wins. Persistent state survives session restarts, enables cross-session workflows, and provides searchable history.
State Source of Truth Contract banners in AI response text are parsed to extract state. Contract detection via regex patterns. MCP tools are authoritative (adv_task_update, etc.). Contract banners are derived views only. ADV wins. Tool-based state is atomic and reliable. Text parsing is fragile and can drift from actual state.
Plugin Architecture 4 modules: types.ts, terminal.ts, contract.ts, index.ts. Event dispatch map pattern. 9 modules: types, schemas, storage (JSON + SQLite), tools (13 MCP tools), validator, archive, events, index. ADV wins. More modular with clear separation of concerns. Each module is independently testable.
Runtime Validation Zod schemas for SDK event validation. 8 schemas defined. Zod schemas for all types + comprehensive validation engine with error codes and suggestions. ADV wins. Richer validation with actionable error messages and fix suggestions.
Test Coverage 51 tests across 3 files. Basic coverage of core functionality. 222 tests across 11 files. Comprehensive coverage including edge cases and error paths. ADV wins. 4x more tests with broader coverage. More confidence in reliability.

Contract & Workflow System

Feature Goost ADV Assessment
Contract Creation /contract or /contract-quick - AI generates contract from user request. Immediate, conversational. /adv-proposal creates scaffold with change.json, proposal.md, initial tasks. Structured, file-based. Goost wins for speed. Immediate contract creation vs multi-step scaffolding. ADV better for compliance.
Contract Immutability Once confirmed, only user can modify/void. "Completed contract = trust. Abandoned = broken trust." Change status transitions: draft → active → archived. Validation gates between states. Tie. Both enforce immutability but differently. Goost is behavioral, ADV is structural.
Validation User confirms contract before locking. No automated validation rules. /adv-validate with 10+ error types, strict mode, detailed suggestions. Gates implementation. ADV wins. Automated validation catches errors before implementation begins.
Task Management Tasks embedded in contract text. No separate task state. adv_task_* tools: list, ready, update, add. Tasks have IDs, status, dependencies. ADV wins. First-class task entities with dependency tracking and status management.
Progress Tracking CONTRACT STATUS block parsed from each response. "Phase: X of Y | Criteria: N/M complete" Tool state is authoritative. Contract banners derived from adv_task_list output. ADV wins. Accurate progress from tool state vs text parsing which can drift.
Completion CONTRACT FULFILLED banner + optional git commit + CHANGELOG update. /adv-archive applies deltas to specs, generates docs, creates archive record. ADV wins. Changes are permanently recorded in specs and archive. Goost has no persistent record.

Overall Verdict

Choose Goost When:
Quick, conversational contracts needed
Single-session tasks that don't need persistence
Lightweight setup is priority
Contract system is sufficient (no spec management)
Using Goost-specific utilities (search, slop-scan, improve)
Choose ADV When:
Long-running projects with persistent requirements
Multiple contributors need shared spec source of truth
Compliance/audit trail requirements
Need searchable specs across large codebase
Want automated validation before implementation
Use Both Together:
ADV for long-term spec management
Goost for immediate task contracts
ADV specs inform Goost contract criteria
Goost contracts for quick wins within ADV workflow
Summary: ADV has superior architecture (tools, persistence, validation) but Goost is faster for simple cases. ADV commands are ports of Goost originals, so command functionality is equivalent. The key difference is how state is managed: Goost parses text, ADV uses tools. For serious spec-driven development, ADV is the clear choice. For quick contracts, Goost remains valuable.
