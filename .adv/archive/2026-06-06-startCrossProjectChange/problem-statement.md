Cross-project change creation (`adv_change_create` with `target_path`) writes the target project's change only to disk and never starts its Temporal change workflow. In `plugin/src/tools/change.ts` (~L787-792) the cross-repo path uses `createLegacyStore(target_path, …)` with an explicit comment that "Temporal runtime is not initialized for external repos in this tool path." It calls `targetStore.changes.create(...)` (disk projection) but never calls `startChangeWorkflow`/`ensureChangeWorkflowStarted` for the target project.

Since the CLI status read is now worker-free but Temporal-Visibility-backed (`rq-statusCliWorkerFree01`), a change with no Temporal workflow has no Visibility search attributes and therefore never appears in `adv status --json` or the launcher's ADV rows. Confirmed live: `reworkRelatedCards` exists on disk in pokeedge (projectId 4d6b589871e3687c746bf043301cfb4ac98ea049) but has no `adv/change/<pid>/reworkRelatedCards` workflow and no `AdvChangeId='reworkRelatedCards'` Visibility entry, so it is invisible in the launcher when pokeedge is selected — even though it was correctly created.

Starting a workflow in the target project is not a Temporal anti-pattern: all ADV projects share one namespace; isolation is by workflowId/taskQueue; `client.workflow.start` enqueues without requiring a live worker; `ensureChangeWorkflowStarted` already sets start-time search attributes (AdvChangeId/Status/CurrentGate/AffectedProjects), is idempotent, and hydrates from disk. The only rule is to never synchronously query the target workflow from the source process (that needs the target worker) — the Visibility read already avoids queries.

This change makes cross-project create start the target change workflow so the change is immediately visible to live status/launcher, and reconciles already-created disk-only changes so existing orphans like reworkRelatedCards become visible.
