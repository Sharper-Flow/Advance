# Temporal Migration — Tooling Gaps Found During 2026-05-02 Pokeedge Cleanup

This is a working notes file capturing tooling weaknesses surfaced while cleaning up corrupted ADV state in `pokeedge`, `pokeedge-web`, and the advance plugin itself. The corruption pattern is consistent with incomplete or under-tested code paths from the Temporal migration era.

## Severity classification

| Sev | Meaning |
|---|---|
| P0 | Bug actively producing corruption / data loss / wrong state |
| P1 | Tool fails on realistic inputs without clear recovery path |
| P2 | Workflow gap (no tool exists for needed operation) |
| P3 | UX / discoverability friction; agent-burdening |

---

## Findings

### F1 — `adv_change_archive` silently stuck on “gates incomplete” when gates are complete (P0)
**Symptom:** `showtranslationpanelprice` had all 7 gates ✓ done with user `sign off` recorded, but multiple archive attempts failed. Gate notes literally read:
> "Re-confirming execution gate after workflow repair because archive tool reported stale incomplete gates despite gate status showing complete."

User had to abandon proper archive flow; agent worked around via direct disk surgery (status:draft → status:archived + dir move). Workflow has remained stuck in `draft` for ~1 day with all gates ✓.

**Root cause hypothesis:** Temporal workflow state diverged from disk state. Archive tool reads from Temporal, gate-status tool reads from disk (or merged view). When they disagree, archive blocks but provides no diagnostic.

**Improvements needed:**
- `adv_change_archive` should detect Temporal/disk divergence and refuse with a SPECIFIC error pointing to `adv_workflow_repair` instead of generic "gates incomplete"
- Add divergence detection to `adv_change_show` so the snapshot warns when Temporal and disk views differ
- `adv_change_archive` should accept a `--repair-and-retry` flag (with approval evidence) for the obvious next step
- Add explicit test: workflow_repair → archive happy path

### F2 — `adv_archive_sweep_orphans` silently absent from migration (P0)
**Symptom:** Pokeedge had **300 synthetic test changes** (`changeRoundtrip{1..54}`, `gateParity{1..54}`, `reentryParity{1..54}`, `taskParity{1..54}`, `wisdomParity{1..54}`, `parityLegacy*`, `parityTemporal*`) all `status: closed` but with source dirs still in `changes/`. Plus 29 `status: archived` source dirs (archive bundles existed in `archive/`) — pure leaks.

Grand total: 335 leaked source dirs that `adv_archive_sweep_orphans includeClosed: true` would have caught.

These changes were generated by **migration parity testing** (the names confirm: roundtrip/gate/reentry/task/wisdom × 54 + legacy/temporal × 3 each). The migration tests left state behind without cleanup.

**Root cause hypothesis:** Migration tests:
1. Created synthetic changes via `adv_change_create` to verify Temporal parity with legacy storage
2. Closed them via `adv_change_close` (or directly)
3. Did NOT call `adv_archive_sweep_orphans` afterward
4. Test fixtures didn't isolate to a temp project, so they leaked into the real `pokeedge` project state

**Improvements needed:**
- Migration parity tests MUST use isolated project IDs (random UUIDs in temp dirs), never real project state
- Add a test-fixture audit: grep for `adv_change_create` in test files, ensure each is paired with cleanup
- Consider auto-running `adv_archive_sweep_orphans` on session shutdown when `closed_change_count > N` (configurable)
- Document a periodic-maintenance command for users: `/adv-cleanup --sweep-orphans`

### F3 — `adv_workflow_repair` rejects valid disk state with cryptic schema error (P1)
**Symptom:** When importing 9 hand-authored proposal directories (only `change.json` + `proposal.md`), `adv_workflow_repair` failed every one with:
```
Schema validation failed for change "X":
Issues: - deltas: Invalid input: expected record, received undefined
Hint: Ensure the change.json matches the schema.
```

The hand-authored `change.json` was missing: `deltas`, `adv_project_id`, `judgment_calls`, `reentry_history`, `wisdom`. Tool refused even though all of these have safe defaults (`{}`, project-id from current context, `[]`, `[]`, `[]`).

**Improvements needed:**
- `adv_workflow_repair` should be **lenient on import**: fill missing required fields with safe defaults rather than fail
- The "Hint" message should specify WHICH fields are missing AND offer a `--seed-defaults` mode
- Schema for `change.json` should distinguish "produced-by-tool fields" (always present) from "imported-from-disk fields" (may be absent in legacy/external snapshots)
- The hand-authored proposal flow itself isn't documented — what's the supported path for `adv_change_create` from a stub?

### F4 — Mutation tools ignore `target_path` despite cross-project doc claims (P1)
**Symptom:** `ADV_INSTRUCTIONS.md § Cross-Project Coordination` says:
> Mutations: use ADV tools in `temporal-required` mode; target queue must be reachable.
> Untrusted mutation: require `target_confirmed: true` + `confirmationEvidence` citing approval.

But the actual tool schemas for `adv_change_archive`, `adv_change_close`, `adv_change_bulk_close`, `adv_workflow_repair`, `adv_orphan_sweep`, `adv_archive_sweep_orphans`, `adv_temporal_register_search_attributes`, `adv_temporal_worker_restart` do **NOT** expose `target_path`. They hard-bind to current process project ID.

When invoked from wrong project the error is `WorkflowNotFoundError: workflow not found for ID: adv/change/{wrong-project-id}/{change-id}` — confusing, doesn't explain you're in the wrong session.

**Improvements needed:**
- Either: implement `target_path` on mutation tools (matches docs), OR
- Document explicitly which tools are project-bound and why
- When `WorkflowNotFoundError` occurs and the changeId exists in any other project's disk state, suggest: "Found change in project X — switch sessions or use opencode run --dir <repo>"
- Add a project-router layer that auto-spawns a target-project worker when a mutation arrives with `target_path`

### F5 — `adv_status target_path: <other-project>` hangs / times out (P0)
**Symptom:** `adv_status target_path: /home/jrede/dev/pokeedge-web` from advance-plugin session timed out at 10s with `errorClass: ToolExecutionTimeout`. Tool description says it should give project overview.

Workaround used: `adv_change_list target_path: ...` (which DOES work via `disk-snapshot` mode).

**Improvements needed:**
- `adv_status` must support `target_path` in `disk-snapshot` mode just like `adv_change_list` does
- If `disk-snapshot` mode genuinely cannot satisfy `adv_status`, the tool should fail fast with "use adv_change_list for cross-project read" instead of timing out
- Test coverage: `adv_status` with `target_path` to a different valid project root

### F6 — Concurrent-session git thrashing (P0)
**Symptom:** With 7 OpenCode sessions running concurrently in pokeedge-web, an unrelated `git commit` from one session ran `git reset HEAD` + `git checkout -- .` to clean its working tree, **silently restoring the legacy `.adv/` files I had just deleted**. Reflog showed the reset between my delete and the unrelated commit.

This isn't ADV's bug per se, but ADV doesn't warn about the danger of multiple sessions per repo.

**Improvements needed:**
- ADV should emit `[ADV:WARN]` at session start if multiple OpenCode processes hold the same project's CWD
- Document the concurrent-session hazard in `ADV_INSTRUCTIONS.md`
- Optional: ADV plugin could grab a per-project file lock at session start (advisory; opt-in)

### F7 — In-repo `.adv/{changes,archive,db,agenda*}` lingered after Temporal migration (P1)
**Symptom:** Pokeedge-web had 4.5MB of legacy in-repo state under `.adv/` even though Temporal migration moved authoritative state external. The in-repo files were stale, partially git-tracked, and confused multiple sessions (some sessions still wrote to them).

**Improvements needed:**
- Migration script should have either: (a) deleted the legacy in-repo paths during the cutover commit, or (b) added them to `.gitignore`
- Add a `/adv-status` warning: "in-repo legacy state detected at .adv/changes/ — run /adv-migrate-cleanup"
- Provide an `adv_migrate_cleanup` tool that does the safe deletion + git commit

### F8 — No project-isolation guarantee for hand-authored proposals (P2)
**Symptom:** A user (or earlier agent) hand-authored 9 proposal directories at `pokeedge-web/.adv/changes/{name}/{change.json,proposal.md}` with `target_path: /home/jrede/dev/pokeedge` (different repo). They sat unrecognized for 12+ hours.

ADV has no documented "import from disk to external state" workflow. `adv_workflow_repair` works on existing-but-broken changes; it isn't an importer.

**Improvements needed:**
- Add an `adv_change_import` tool: takes `path` to a dir containing `change.json` + `proposal.md` and reseeds it into Temporal + external state (with field defaults — see F3)
- Document the import path in SETUP.md
- Decide & document: are hand-authored proposals supported? If yes, what's the canonical schema?

### F9 — `adv_change_list status: "active"` filter excludes `draft` changes (P2)
**Symptom:** During verification I asked an opencode subagent to run `adv_change_list status: "active"` after binding the orphans. It returned 0 changes despite 16 being on disk. All changes were `status: "draft"`, but the user-visible filter `"active"` is widely used in agent prompts to mean "currently in flight".

The schema enum `"draft" | "pending" | "active" | "archived" | "closed"` is leaky — agents prompt with the colloquial meaning of "active".

**Improvements needed:**
- Define a `status: "in-flight"` shortcut that means `draft + pending + active` (the union most agents want)
- Or: rename `"draft"` to `"active-draft"` and `"active"` to something more specific
- Document the difference in `ADV_INSTRUCTIONS.md`

### F10 — `opencode run --agent build` extremely slow on simple ADV verification prompts (P2)
**Symptom:** A subsession via `opencode run --dir <pokeedge> --agent build "Run adv_orphan_sweep dryRun:true and adv_status. Print verbatim output."` timed out at 120s with **zero tool calls executed**, just an agent banner. Agent reasoning eats budget before any tool call lands.

**Improvements needed:**
- ADV should expose a `--exec-tools-only` mode (or a non-LLM CLI shim) for cross-project ops where you just want to call N tools and exit
- Alternatively: an `opencode tool <name> --dir <dir> --args <json>` that calls a single MCP tool and exits without any LLM round-trip
- This is the difference between needing a 30s call vs. a 5min agent loop

### F11 — Synthetic-test-leak detection has no agenda-level signal (P3)
**Symptom:** 300 synthetic changes sat in pokeedge for weeks/months. No alert, no `[ADV:WARN]`, no `/adv-status` flag. The user only noticed when state corruption symptoms surfaced.

**Improvements needed:**
- Add a periodic check (session-start or `/adv-status`): if `closed_count > active_count * 5`, emit `[ADV:WARN]` suggesting `adv_archive_sweep_orphans includeClosed: true`
- Add `_healthSnapshot` field to `adv_status` output: `{ leaked_source_dirs: N, closed_change_dirs: N, archive_dirs: N }`

### F12 — `adv_change_show` with stuck change can return `canArchive: true` but archive can't run (P0)
**Symptom:** `adv_change_show changeId: showtranslationpanelprice` and `adv_gate_status` both reported `canArchive: true` and `nextGate: null` (all gates done). But `adv_change_archive` failed. The state-validation tool and the action-tool disagree.

**Improvements needed:**
- `canArchive: true` MUST imply `adv_change_archive` will succeed. If they can disagree, that's a contract violation.
- Add an integration test: for any change where `canArchive == true`, archive must succeed (or both must agree it cannot).

### F13 — Worker-restart and orphan-sweep are heavyweight even when current state is fine (P3)
**Symptom:** When debugging F1 / F12, the only suggested fixes were `adv_temporal_diagnose` → `adv_temporal_reconnect` → `adv_temporal_worker_restart` → `adv_orphan_sweep` → `adv_workflow_repair`. None of these are diagnostic-only; the recovery menu is heavy.

**Improvements needed:**
- Add `adv_change_diagnose changeId: <id>` — read-only deep inspection of one change's Temporal state vs disk state, with a recommended specific fix
- Surface fix-grade vs. nuclear-grade options separately

### F14 — Tool description doesn't say `dryRun: true` is safe to run anytime (P3)
**Symptom:** Throughout this session I ran several `dryRun: true` ops, but the docs / tool descriptions don't loudly state "always safe to run, no mutation, useful for inspection". Agents (especially this one) are correctly cautious about anything that says "sweep" or "repair", and may avoid even dry-runs.

**Improvements needed:**
- Tool description boilerplate: `"With dryRun: true, this tool is read-only and safe to invoke without approval."`
- Surface `dryRun` defaults loudly
